# PRD – Diseño de Mockup Moderno para Plataforma ITSM

## 1. Nombre del producto

**Project codename:** NovaDesk ITSM

## 2. Propósito del documento

Definir de manera funcional, visual y experiencial los requerimientos para diseñar un **mockup moderno de una plataforma ITSM** con enfoque **enterprise SaaS**, inspirado en patrones UX/UI exitosos de herramientas como Freshservice y Zendesk, priorizando:

* simplicidad visual
* operación rápida
* baja fricción
* navegación clara
* alta densidad de información bien organizada
* percepción premium, moderna y confiable

## 3. Objetivo del producto

Diseñar un mockup de plataforma ITSM que permita visualizar una solución de service desk moderna para organizaciones que necesitan gestionar:

* incidentes
* requerimientos de servicio
* problemas
* cambios
* activos
* base de conocimiento
* SLAs
* automatizaciones
* analítica operacional

El mockup debe transmitir una herramienta:

* intuitiva
* elegante
* altamente usable
* escalable
* enterprise-ready
* minimalista pero funcional

## 4. Problema a resolver

Las herramientas ITSM tradicionales suelen presentar uno o más de estos problemas:

* interfaces recargadas
* exceso de tablas sin jerarquía visual
* navegación compleja
* demasiados clics para acciones frecuentes
* mala separación entre operación diaria y configuración
* experiencia visual poco moderna
* baja adopción por parte de agentes y usuarios finales

Se busca un diseño que elimine esa fricción y ofrezca una experiencia comparable a plataformas modernas de servicio que centralizan tickets, contexto, automatización y canales en una sola vista. Freshservice resalta precisamente una vista unificada para tickets, automatizaciones drag-and-drop, omnicanalidad y autoservicio; Zendesk enfatiza la operación desde un único workspace de agente. ([Freshworks][1])

## 5. Visión del producto

Crear una plataforma ITSM cuya experiencia visual combine:

* la limpieza y estructura moderna de Zendesk
* la sensación de productividad y operación IT de Freshservice
* una estética minimalista tipo SaaS premium
* flujos optimizados para agentes, supervisores, administradores y usuarios finales

## 6. Alcance del mockup

El mockup debe cubrir como mínimo las siguientes áreas principales:

1. Login / acceso
2. Dashboard principal
3. Listado de tickets
4. Vista detalle de ticket
5. Portal de autoservicio
6. Base de conocimiento
7. Catálogo de servicios
8. Gestión de activos / CMDB ligera
9. Automatizaciones / workflow builder
10. Reportes y analítica
11. Configuración administrativa
12. Centro de notificaciones
13. Vista de SLA y prioridades
14. Bandeja omnicanal / entradas
15. Vista mobile responsive conceptual

## 7. Objetivos de diseño

El diseño debe lograr:

### Objetivos primarios

* Reducir carga cognitiva
* Permitir entender el estado operativo en menos de 5 segundos
* Facilitar acciones frecuentes con 1 a 2 clics
* Mejorar la percepción de orden, velocidad y control
* Hacer que la plataforma se vea premium y confiable

### Objetivos secundarios

* Facilitar escalabilidad futura
* Permitir adaptación a multicanal
* Servir como base visual para MVP o desarrollo real
* Soportar temas claro y oscuro

## 8. Usuarios objetivo

### 8.1 Agente de soporte

Responsable de atender tickets, responder usuarios, actualizar estados, escalar casos y cumplir SLA.

**Necesidades**

* ver cola de tickets claramente
* identificar prioridad rápido
* acceder a contexto del usuario
* ejecutar acciones rápidas
* responder desde una vista única

### 8.2 Supervisor / líder de mesa

Responsable de monitorear operación, carga del equipo, SLAs, backlog, asignaciones y desempeño.

**Necesidades**

* ver métricas en tiempo real
* detectar cuellos de botella
* reasignar tickets
* monitorear agentes
* ver tickets en riesgo

### 8.3 Administrador ITSM

Configura formularios, automatizaciones, SLAs, catálogos, roles y reglas.

**Necesidades**

* interfaz ordenada
* configuración simple
* separación clara entre operación y administración
* control de permisos
* visibilidad del impacto de cambios

### 8.4 Usuario final / empleado

Solicita ayuda, consulta artículos, hace seguimiento a solicitudes.

**Necesidades**

* portal simple
* búsqueda clara
* autoservicio
* seguimiento intuitivo
* sensación de rapidez y transparencia

## 9. Principios UX/UI

El diseño deberá seguir estos principios:

### 9.1 Claridad primero

La interfaz debe priorizar comprensión inmediata sobre decoración excesiva.

### 9.2 Minimalismo funcional

Pocos elementos visuales, pero altamente útiles.

### 9.3 Jerarquía visual fuerte

Cada pantalla debe responder visualmente:

* qué es más importante
* qué requiere acción
* qué es secundario

### 9.4 Consistencia

Mismos patrones para:

* botones
* filtros
* estados
* badges
* tabs
* modales
* tablas
* formularios

### 9.5 Productividad operacional

El diseño debe sentirse hecho para trabajar rápido.

### 9.6 Contexto unificado

La vista de ticket debe centralizar:

* conversación
* datos del solicitante
* historial
* SLA
* activos relacionados
* acciones rápidas

### 9.7 Diseño calmado

Evitar ruido visual; usar aire, espaciado, contraste sutil y componentes limpios.

## 10. Referencias de experiencia deseada

La solución debe inspirarse en patrones visibles en productos actuales de Freshservice y Zendesk:

* **vista unificada de ticket**
* **omnicanalidad**
* **autoservicio**
* **automatizaciones**
* **configuración simple**
* **analítica operativa**
* **workspace limpio para agentes**
* **administración sin fricción** ([Freshworks][1])

* espacio para IA copilot conceptual

## 14. Flujos principales a mockear

### Flujo 1 – Agente atiende ticket

1. entra al dashboard
2. ve tickets críticos
3. abre ticket
4. revisa contexto
5. responde
6. cambia estado
7. asigna o resuelve
8. registra nota interna

### Flujo 2 – Usuario crea solicitud

1. entra a portal
2. busca ayuda
3. no encuentra artículo
4. crea solicitud
5. llena formulario
6. recibe confirmación
7. hace seguimiento

### Flujo 3 – Supervisor revisa desempeño

1. entra a dashboard
2. filtra por equipo
3. detecta SLA en riesgo
4. abre cola crítica
5. reasigna tickets

### Flujo 4 – Admin crea automatización

1. entra a admin
2. abre workflow builder
3. configura trigger
4. agrega condiciones
5. define acciones
6. guarda y activa

## 21. Criterios de éxito del mockup

El mockup será exitoso si:

* se percibe visualmente al nivel de un SaaS moderno enterprise
* un usuario entiende la estructura general en menos de 1 minuto
* la vista de ticket transmite productividad y control
* el dashboard comunica estado operativo de forma inmediata
* el portal de autoservicio se ve amigable y claro
* el sistema luce consistente en todos sus módulos
